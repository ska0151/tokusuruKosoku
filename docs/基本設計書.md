# 基本設計書（BDD）

## 1. はじめに

本文書は、首都圏高速IC比較アプリ「トクする高速」の基本設計を定めるものです。
RDD（要件定義書）で記載された要件を実現するため、**無料・低コストで実現可能な技術スタック**を採用した実装方針を示します。

## 3. 無料化検討と技術選定

### 3.1 コスト課題と対策

#### 課題 1: Google Maps API の高コスト
**RDD での計画**
- Directions API を主軸に最大 5 通りのルート計算を並列実行
- 月間コスト：最悪 50,000+ リクエスト × $5/1000 = 数百〜数千円規模

**採用案**
- **OSRM（Open Source Routing Machine）** を自前もしくはパブリックインスタンスで利用
  - 完全無料、オープンソース
  - OpenStreetMap データを使用
  - Google Maps と同等の精度を提供可能
  - インスタンス: https://router.project-osrm.org （パブリック API も無料）
  
**メリット**
- API 利用料金ゼロ
- キャッシュ効果が高い（同じ IC ペアの重複検索が多い）

**デメリット**
- 自前ホスティングの場合、月間コスト若干あり（後述）
- リアルタイム渋滞情報は別途取得が必要

---

#### 課題 2: リアルタイム渋滞情報

**対策**
1. **第一段階（MVP）**: 渋滞情報なし、距離＋標準所要時間で計算
2. **第二段階**: 以下いずれかから無料/低コスト情報を取得
   - NEXCO の公開データ API（時間帯別混雑情報）
   - OpenWeather の道路情報（限定的）
   - ユーザー投稿型の遅延情報

**推奨**: MVPは固定所要時間で開始し、ユーザーフィードバック後に段階導入

---

#### 課題 3: インフラコスト
**採用案**

| サービス | 対象 | 無料枠 | 備考 |
|---|---|---|---|
| **Vercel** | フロントエンド（Next.js） | 無制限 | edge functions 12.5万リクエスト/日まで無料 |
| **Railway** | バックエンド + DB | $5/月クレジット相当 | PostgreSQL, Redis も含む | 
| **Fly.io** | バックエンド（代替） | $3/月クレジット | 低スペックアプリ向け |
| **Render** | バックエンド（代替） | 無料（スリープあり） | 低トラフィック向け |
| **Redis Cloud** | キャッシュ | 30 MB 無料 | 小規模用、大規模なら Railway 推奨 |

**推奨構成**
- **フロント**: Vercel（ほぼ無制限）
- **バック + DB**: Railway 課金枠内（月額数百~千円程度）
- **キャッシュ**: 小規模なら Redis Cloud、大規模なら Railway に統一

### 4.2 データモデル

#### IC マスタ
```sql
CREATE TABLE ics (
  id VARCHAR(20) PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  hiragana VARCHAR(100),
  operator VARCHAR(50),  -- NEXCO-EAST, NEXCO-CENTRAL, NEXCO-WEST
  input BOOLEAN,
  output BOOLEAN,
  location GEOMETRY(Point, 4326),  -- PostGIS
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);

CREATE TABLE ic_constraints (
  id SERIAL PRIMARY KEY,
  ic_id VARCHAR(20),
  constraint_type VARCHAR(50),  -- "no_exit_right", "no_entry_certain_hours"
  description VARCHAR(200),
  FOREIGN KEY (ic_id) REFERENCES ics(id)
);
```

#### 料金セグメント
```sql
CREATE TABLE toll_segments (
  id SERIAL PRIMARY KEY,
  from_ic_id VARCHAR(20),
  to_ic_id VARCHAR(20),
  distance DECIMAL(8, 2),
  base_toll INT,  -- 基本料金（円）
  valid_from DATE,
  valid_until DATE,
  updated_at TIMESTAMP,
  FOREIGN KEY (from_ic_id) REFERENCES ics(id),
  FOREIGN KEY (to_ic_id) REFERENCES ics(id)
);
```

#### ETC割引ルール
```sql
CREATE TABLE discount_rules (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100),
  rule_type VARCHAR(50),  -- "percentage", "fixed", "cap"
  discount_percent DECIMAL(5, 2),
  condition_days VARCHAR(20),  -- "weekday", "weekend", "all"
  condition_time_start INT,  -- 0-2359
  condition_time_end INT,
  apply_from DATE,
  apply_until DATE
);
```

#### API リクエストキャッシュ
```sql
CREATE TABLE route_cache (
  id SERIAL PRIMARY KEY,
  origin_lat DECIMAL(10, 8),
  origin_lng DECIMAL(11, 8),
  destination_lat DECIMAL(10, 8),
  destination_lng DECIMAL(11, 8),
  vehicle_type VARCHAR(20),
  has_etc BOOLEAN,
  route_result JSONB,
  cache_hit_count INT DEFAULT 1,
  created_at TIMESTAMP,
  expired_at TIMESTAMP,
  INDEX idx_search (origin_lat, origin_lng, destination_lat, destination_lng)
);
```

## 5. UI/UX 設計概要

### 5.1 画面フロー

```
1. ホーム画面
   ├ 現在地取得（ボタン大きく）
   → 目的地入力フィールド（キーワード or ピン配置）
   → 検索ボタン

2. IC 選択画面（自動提案）
   ├ 周辺 IC リスト表示
   ├ 各 IC への地図表示、アクセス時間
   → 確定ボタン

3. ルート比較画面（メイン）
   ├ 3 案横スクロール表示
   │ ├ 【最速】所要時間 XX 分 / 料金 ¥XX,XXX
   │ ├ 【バランス】所要時間 XX 分 / 料金 ¥XX,XXX
   │ └ 【最安】所要時間 XX 分 / 料金 ¥XX,XXX
   ├ 差分表示（"△600円 / -5分" など）
   ├ 地図表示（ルート、IC 位置）
   → 各ルートをタップで詳細
   → 「カーナビに転送」ボタン

4. ルート詳細画面
   ├ IC 属性表示
   ├ セグメント別料金内訳
   ├ 割引適用状況
   → 確定・キャンセル

5. 外部ナビアプリ連携
   ├ 免責事項同意
   → Google Maps/Apple Maps 起動
```

### 5.2 デザイン方針

- **モバイルファースト**: 画面サイズ 320px 以上を想定
- **大きなボタン**: 車内操作を考慮し 48px 以上
- **高コントラスト**: 昼間の日光読みやすさを重視
- **シンプルな表示**: 複雑な説明は避け、アイコン＋金額で直感的に
- **アニメーション最小**: 車内酔い軽減

### 5.3 アクセシビリティ / 安全性

- **走行中の制限**: 位置情報更新 5 秒ごと → 確定後は操作最小化
- **音声ガイド**: Optional（iOS VoiceOver, Android TalkBack対応）
- **免責事項**: 「安全な場所で確認してください」と初回必ず表示

## 7. セキュリティ・コンプライアンス

### 7.1 セキュリティ対策

| 対策項目 | 実装方法 |
|---|---|
| **通信暗号化** | HTTPS + TLS 1.2 以上（Vercel / Railway で自動） |
| **位置情報保護** | 検索時のみ使用、サーバー側で即時削除、ログ記録なし |
| **API 認証** | 初期段階では認証なし、スケール後 API キー or OAuth2 |
| **レート制限** | 1 IP あたり 10 リクエスト/分 |
| **CORS 設定** | フロント許可ドメインのみ |
| **SQL インジェクション対策** | Prepared Statement 必須（Knex.js など） |

### 7.2 法規制対応

| 項目 | 対応 |
|---|---|
| **ながら運転対策** | 走行中の詳細操作制限、初回安全確認画面 |
| **個人情報保護** | 位置情報非永続化、プライバシーポリシー掲示 |
| **利用規約** | 免責事項明記（ナビアプリと同等） |

## 9. リスク・課題と対策

| リスク | 発生確率 | 影響度 | 対策 |
|---|---|---|---|
| **OSRM 精度不足** | 中 | 中 | 初期段階で複数 IC ペアで検証、必要なら Google Maps Directions 併用 |
| **NEXCO 料金データ入手困難** | 低 | 高 | 公開情報のスクレイピング or 手入力、段階リリース |
| **渋滞情報の遅延** | 低 | 低 | MVP は固定所要時間、後日渋滞 API 統合 |
| **API 利用料急増** | 低 | 中 | OSRM + Redis キャッシュで対策済み |
| **ユーザー採集困難** | 中 | 中 | SNS / オタクコミュニティ積極展開 |
