# Requirements Definition Document (RDD)

## プロジェクト概要

### システム化の背景
首都圏の高速道路網は極めて複雑であり、入口ICを一つ変えるだけで料金が大きく変動する特性があります。しかし、既存のカーナビや地図アプリの多くは「最速ルート」を優先し、数分の時間増加と引き換えに得られる数百円の節約（コストパフォーマンス）を提示する機能が不足しています。特に物価高騰の影響もあり、日常的に高速を利用するドライバーの間で「賢い節約」へのニーズが高まっています。

### 目的
ドライバーが現在地周辺の複数のIC入口を起点とした「料金」と「時間」のトレードオフを瞬時に比較し、納得感のあるルート選択を行える環境を提供すること。

### 期待効果
*   ユーザーの通行料支出の最適化。
*   渋滞が激しい都心部ICを避け、郊外ICへ誘導することによる交通分散への寄与。
*   「いくら得したか」を可視化することによる、運転の心理的満足度の向上。

### 対象範囲
*   首都圏（東京・神奈川・千葉・埼玉）を中心とした高速道路および周辺一般道。
*   乗用車（軽自動車・普通車）を主対象とした料金シミュレーション。

## 業務要件

### 現行業務フロー
1.  ドライバーが目的地をカーナビに入力する。
2.  ナビが提示する「推奨（最速）」または「料金優先（全下道など極端なもの）」から選択する。
3.  他のICから乗った場合の差額を知るには、経由地を個別に入力し直して再計算する必要がある。

### 新業務フロー
1.  アプリを起動し、目的地を選択する。
2.  システムが周辺15km圏内の有効なICを自動抽出し、3つの比較案（最速・バランス・最安）を提示する。
3.  ユーザーが比較案をタップし、各ICへのアクセスと料金差を確認してルートを決定する。
4.  決定したIC情報を外部ナビアプリに転送し、走行を開始する。

### 業務ルール
*   渋滞情報は常に最新（5分更新程度）のものを参照し、到着予想時刻に反映させる。
*   ETC割引は検索時の曜日・時間帯に基づき、適用可否を自動判定する。

### 承認プロセス
本アプリはBtoCのツールであるため、ユーザー側の承認プロセスは存在しませんが、外部ナビアプリへの遷移時に「安全のための前方注視」を促す免責事項の同意プロセスを設けます。

## 機能要件

### システムが提供すべき機能の詳細仕様
*   **周辺ICスキャン機能**: 現在地から走行可能な入口ICを最大5件特定。
*   **トクする比較表示**: 標準ルートに対する「差額」と「追加時間」の算出。
*   **IC属性表示**: 入口・出口の種別、および首都高特有の右入り口等の制約情報の提示。

### 入出力項目
*   **入力**: 目的地（キーワード/マップピン）、車種区分、ETC有無。
*   **出力**: ルート別料金、ルート別所要時間、IC位置情報、節約額、比較地図。

### 処理内容
1.  目的地確定後、Directions APIにより周辺IC経由のルートを並列計算。
2.  取得した距離・区間情報を料金エンジンに照会。
3.  割引計算ロジックを通した後、ユーザー設定に最適な3案を抽出・ソート。

#### ※バックエンドの役割
**複雑な料金ロジックの集中管理:**

 - 距離制、区間定額、各種ETC割引が絡み合う複雑な計算をサーバー側エンジンで一括処理。
 - 料金改定時もサーバー側の修正のみで即時反映を実現。

**APIの集約とコスト最適化:**

 - Google Maps Platform等の外部APIをバックエンドで集約。

 - キャッシュ戦略（Redis）により、同一ルート検索の再計算を抑制し、API利用料を削減。

**重量級の並列計算処理:**

 - 最大5通りのルート探索と料金照会をサーバーリソースを用いて並列実行。

 - フロントエンドには整理済みの「3案」のみを返し、低スペック端末でも高速レスポンスを維持。

**静的データの保持と配信:**

 - ICマスタ、制約情報、座標等の膨大なデータをDB管理し、必要な分だけを配信することでアプリの軽量化を図る。

## 非機能要件

### 性能要件
*   目的地入力から比較案提示までのレスポンスタイムを3秒以内とする。

### 可用性要件
*   24時間365日の稼働（ただし、参照する外部APIのメンテナンス時間を除く）。

### セキュリティ要件
*   通信のSSL/TLS化。
*   位置情報の取り扱いは検索時のみとし、サーバー側への永続的な保存は行わない。

### 操作性要件
*   「車内での即時判断」を支えるため、ボタンサイズを大きくし、コントラストの高いUIを設計する。

## 技術要件

### システム構成
フロントエンドとAPIサーバーを分離したSPA/PWA構成。

### 開発言語
*   **フロントエンド**: TypeScript / React (Next.js)。
*   **バックエンド**: Node.js または Python (FastAPI)。

### データベース
*   **ICマスタ情報および料金テーブル保持用**: PostgreSQL。
*   **キャッシュ用**: Redis。

### インフラ要件
*   **クラウドプラットフォーム**: AWS または Google Cloud。
*   **地図・経路基盤**: Google Maps Platform。

---

## 制約条件

### 予算制約
Google Maps APIの利用料を無料枠＋αの範囲に収めるための計算回数制限。

### スケジュール制約
行楽シーズン（GWや夏休み）前のリリースを目指した短期開発。

### 技術制約
スマートフォンのブラウザ（Safari/Chrome）で動作するPWAとし、各OSのアプリストア審査に依存しない配信形態とする。

### 法的制約
道路交通法（ながら運転）への配慮。走行中の操作制限または注意喚起の徹底。
